# 알림 데이터 구조

알림 시스템은 **규칙 → 발생 이벤트 → 채널별 전달 시도**의 3계층으로 구성한다. 기존 웹 푸시 테이블과 API는 유지하고, 새 구조를 추가하여 하위 호환성을 보장한다.

## `notification_rules`

사용자가 설정한 알림 의도를 저장한다.

- 알림 종류: 시간 기반, 시간작업, 기간형 작업, AI 추천, 반복, 요약, 취침·기상, 일반 알람
- 스케줄: 1회, 매일, 매주, 매월, 간격, 앱 데이터에서 계산
- 전달 채널: 앱 내부, 웹 푸시, 네이티브 푸시, 카카오톡, Apple Watch
- 사전 알림, 소리, 재알림, 타임존 설정
- 특정 일정·작업과 연결할 수 있는 `source_type`, `source_id`

## `notification_events`

규칙, 일정, 작업, AI가 생성한 실제 발생 1회를 저장한다. 기존 이벤트 테이블에 규칙, 우선순위, 채널, 소리, 만료 시각, 채널 공통 payload를 추가한다.

규칙에서 생성된 이벤트는 `rule_id`로 연결되며, 기존 일정 이벤트는 `rule_id` 없이도 계속 동작한다.

## `notification_delivery_attempts`

하나의 이벤트를 각 채널로 보낸 결과를 저장한다.

- 상태: 대기, 발송, 전달 확인, 실패, 건너뜀
- 제공자 메시지 ID와 오류 정보
- 이벤트·채널·대상 조합의 중복 발송 방지

기존 `notification_event_deliveries`, `native_notification_event_deliveries`는 현재 발송 로직을 깨뜨리지 않기 위해 유지한다. 추후 채널 어댑터를 전환할 때 공통 전달 기록을 사용한다.

## 보안

- 모든 사용자 데이터 테이블에 RLS를 적용한다.
- 인증 사용자는 `auth.uid() = user_id`인 데이터만 접근한다.
- Data API 접근은 명시적 `GRANT`와 RLS를 함께 사용한다.
- 채널 스키마에 비밀키나 원문 인증 정보를 저장하지 않는다.
